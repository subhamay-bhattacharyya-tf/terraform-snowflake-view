# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this module does

This is a **Terraform module** that creates and manages **Snowflake views** — both standard and secure — using infrastructure as code.

The module is a single, flat Terraform module rooted at the repository root (`main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`). It accepts a single map-based input (`views`) and exposes maps of resource attributes as outputs, allowing one module call to provision many views via `for_each`.

The module manages only `snowflake_view` resources. Grants and other access-control concerns are intentionally out of scope and should be managed by separate, composable modules to prevent drift and align with Snowflake's privilege model.

bash install-tools.sh --tools=terraform,tflint,trivy  # install subset

# Run pre-commit hooks

## Examples

The `examples/` directory contains end-to-end, runnable configurations that exercise the module against real Snowflake views. Each example is a self-contained Terraform configuration that calls the root module via a relative `source = "../.."` reference, declares its own `terraform` and `provider` blocks, and uses the module's `views` map input.

Each example must:

- Use the module's `views` map-based input shape — never inline `snowflake_view` resources
- Reference real Snowflake objects (database, schema, base tables) that a Terratest run can create or assume exist
- Include a short `README.md` explaining what the example demonstrates and the Snowflake objects it provisions
- Be validated by the `examples-validate` CI step (`terraform init -backend=false && terraform validate`)

### `basic/`

A single **standard view** built over a base table — the minimum viable usage. Demonstrates:

- Calling the module with a single-entry `views` map
- Setting `is_secure = false` (or omitting it to use the default)
- Providing a SQL `statement` that selects from a base table
- Wiring `database` and `schema` inputs to a target Snowflake schema

Use this as the reference for new users learning the module's input shape.

### `secure-view/`

A **secure view** for restricted data exposure — demonstrates the security-oriented use case. Demonstrates:

- Setting `is_secure = true` to mark the view as secure (hiding the underlying SQL definition from non-owners)
- Selecting a curated subset of columns suitable for downstream consumers
- Adding a meaningful `comment` describing why the view is secure and which roles should consume it

Use this as the reference when exposing PII, financial, or otherwise sensitive data through views.

#### Sample `views` map

```hcl
module "snowflake_view" {
  source = "../.."

  views = {
    customer_pii_secure = {
      name      = "CUSTOMER_PII_SECURE_VW"
      database  = "ANALYTICS_DB"
      schema    = "CONSUMPTION"
      is_secure = true

      statement = <<-SQL
        SELECT
          cust_key,
          name,
          nation_name,
          mkt_segment
        FROM ANALYTICS_DB.CLEAN.CUSTOMER_CLEAN_DT
      SQL

      comment = "Secure view exposing curated customer attributes; consumed by ANALYST_ROLE only. Underlying SQL hidden from non-owners."
    }
  }
}
```

The map key (`customer_pii_secure`) is a logical Terraform identifier used in `for_each` and in output maps; the actual Snowflake view name (`CUSTOMER_PII_SECURE_VW`) lives inside the object as the `name` field. This separation is intentional — it lets the same Terraform identifier survive a Snowflake-side rename without forcing a destroy/create.

## Utility Scripts

The `utils/` directory contains helper scripts used by both local development and CI. Anything reusable across `pre-commit`, `make` targets, and GitHub Actions belongs here — not at the repo root and not duplicated inside individual workflows.

All scripts must:

- Be runnable from the repository root (`bash utils/<script>.sh` or `python utils/<script>.py`)
- Be idempotent — running them twice in a row produces no diff on a clean tree
- Exit non-zero on failure so they can gate CI steps and pre-commit hooks
- Reference Snowflake views in any user-facing output (no leftover references to DynamoDB, S3, etc.)

### `generate-docs.sh`

Runs `terraform-docs` against the root module and rewrites the auto-generated **Inputs**, **Outputs**, and **Resources Created** tables in `README.md`. Should be invoked:

- Locally before committing any change to `variables.tf`, `outputs.tf`, or `main.tf`
- By the `pre-commit` hook chain
- By a CI doc-drift check that fails the build if `README.md` is out of sync with the module

After this script runs, `align-md-tables.py` should be run to re-pipe-align any tables that `terraform-docs` regenerated.

### `lint.sh`

Wraps **tflint** and **trivy** to lint the root Terraform module and scan it for security issues. Runs:

- `tflint --recursive` with the project's `.tflint.hcl` configuration — catches Snowflake provider misuse, unused variables, deprecated syntax
- `trivy config .` — scans for misconfigurations (e.g. views that should be secure but aren't, missing comments on production resources)

The script exits non-zero on any finding so it can gate CI. Both tools are installed by `install-tools.sh`.

### `align-md-tables.py`

Python auto-aligner that pads every GFM table cell in `README.md` to its column's maximum width, enforcing **markdownlint MD060** (`table-column-style: aligned`). Should be invoked:

- Whenever `generate-docs.sh` regenerates inputs/outputs tables
- Whenever a heading or table is hand-edited in `README.md`
- By the `pre-commit` hook chain so misaligned tables never reach CI

The script collects every `|` position per row and asserts they line up vertically across the header, separator, and body rows — the same structural check described under the README conventions.

### `update-badge.sh`

Updates the shields.io custom-endpoint gist JSON file (`terraform-snowflake-view.json`) hosted as a secret gist. Used to keep the README's coverage / test-status / version badge in sync with the latest CI run. Should be invoked:

- By the `terratest` CI job after a successful Snowflake views integration test
- By the `semantic-release` job after a new version is published

Requires `gh` CLI authentication and the gist ID configured via the `BADGE_GIST_ID` environment variable.

## Tests

The `tests/` directory contains **Terratest** integration tests written in Go that exercise the module against a real Snowflake account. These are not unit tests — they `terraform apply` real `snowflake_view` resources, assert behavior, and `terraform destroy` on teardown.

Each test file must:

- Live under `tests/` (note: plural) — never `test/`
- Be executable via `go test` from the `tests/` directory
- Use the helpers defined in `helpers_test.go` rather than re-implementing setup/teardown
- Target one of the configurations under `examples/` as its Terraform working directory
- Clean up all Snowflake objects in `defer terraform.Destroy(...)` — failures must not leak views into the target account
- Reference Snowflake views in test names, log lines, and assertion messages (no leftover references to DynamoDB, S3, etc.)

Required environment variables (provided as repo secrets/vars in CI):

- `SNOWFLAKE_ACCOUNT` — target Snowflake account identifier
- `SNOWFLAKE_USER` — service-account user with privileges to create views in the test database
- `SNOWFLAKE_PRIVATE_KEY` — key-pair auth private key (preferred over passwords)
- `SNOWFLAKE_ROLE` — role used for the test session; must have `CREATE VIEW` on the target schema
- `SNOWFLAKE_WAREHOUSE` — warehouse used to compile view definitions during apply

### `snowflake_view_basic_test.go`

End-to-end test for the `examples/basic/` configuration. Covers:

- **Apply** — runs `terraform init` + `terraform apply` against `examples/basic/` and asserts the apply succeeds with zero errors
- **Output assertions** — reads the module's output maps (`view_ids`, `view_fully_qualified_names`, `view_names`, `is_secure`) and asserts each map has exactly the expected keys and that the values are non-empty
- **Snowflake-side verification** — connects directly to Snowflake and asserts the view exists in `INFORMATION_SCHEMA.VIEWS`, that `IS_SECURE = 'NO'` for the standard view, and that the resolved `view_definition` contains the expected `SELECT` columns
- **Idempotency** — runs `terraform plan` after apply and asserts the plan is empty (no drift on a no-op re-run)
- **Destroy** — runs `terraform destroy` in a `defer` block and asserts the view no longer appears in `INFORMATION_SCHEMA.VIEWS`

This test is the reference for what a "passing" module change looks like — any new feature should be exercised by either extending this test or adding a sibling test file alongside a new example.

### `helpers_test.go`

Shared helpers consumed by every test file under `tests/`. Provides the contract that every Snowflake-views test relies on:

- **`buildTerraformOptions(t, exampleDir)`** — constructs a `*terraform.Options` pointed at the given example directory, wired with the `SNOWFLAKE_*` environment variables and a unique per-run resource suffix so parallel CI runs don't collide on view names
- **`newSnowflakeClient(t)`** — opens an authenticated `*sql.DB` connection to Snowflake using the same credentials Terraform uses, scoped to the test role and warehouse, with a context-aware timeout
- **`assertViewExists(t, db, fullyQualifiedName)`** — queries `INFORMATION_SCHEMA.VIEWS` and asserts the view is present; fails fast with a descriptive message if not
- **`assertViewIsSecure(t, db, fullyQualifiedName, expected bool)`** — asserts the `IS_SECURE` flag matches the expected value, used to verify the `is_secure` input is honored end-to-end
- **`assertViewDestroyed(t, db, fullyQualifiedName)`** — asserts the view is absent from `INFORMATION_SCHEMA.VIEWS`, used in the post-destroy phase to catch leaked resources
- **`uniqueSuffix(t)`** — returns a short, lowercase, alphanumeric suffix derived from the test name and a random seed; appended to view and database names to keep parallel runs isolated

Helpers must not embed test logic — they are pure setup, teardown, and assertion primitives. Anything specific to a single test (e.g., asserting a particular column list) belongs in that test file, not here.

## Key Conventions

- The module follows the standard flat Terraform layout (`main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`) at the repository root — there are no submodules under `modules/`
- All infrastructure changes go through Terraform — never modify Snowflake views manually via the UI or worksheets
- The module accepts a single `map(object({...}))` input named `views`, consumed via `for_each`
- The map key is a logical Terraform identifier; the actual Snowflake view name is a field inside the object
- Grants are intentionally **not** managed by this module — pair this module with a separate grants module to align with Snowflake's privilege model and avoid drift
- Every example under `examples/` must call the root module via `source = "../.."` and use the `views` map input — no inline `snowflake_view` resources
- Helper scripts live under `utils/` — anything reusable across local dev and CI (doc generation, lint wrappers, validation helpers, badge updates) belongs there, not at the repo root
- Integration tests live under `tests/` (note: plural), not `test/` — Go test files, helpers, and any test fixtures all live in this directory; helpers are pure setup/teardown/assertion primitives, never test logic
- `package.json` and `package-lock.json` must always have their `name` field set to the current repository name (`terraform-snowflake-view`)
- `CONTRIBUTING.md` must always reference the current repository (`terraform-snowflake-view`), including the **Reporting Issues** section which must link to this repository's issues page
- `README.md` must always reflect the current repository (`terraform-snowflake-view`) and the Snowflake object this module provisions (Snowflake views). Every surface that carries the repo name or service name must be kept in sync — specifically:
  - Title (`# Terraform Snowflake View Module`) and the one-line description below it
  - All shields.io badges that interpolate the repo slug (`subhamay-bhattacharyya-tf/terraform-snowflake-view`) — leftover slugs from the upstream template (e.g. `terraform-aws-s3`, `terraform-aws-dynamodb`) must be swapped
  - The shields.io custom-endpoint badge, which points to a gist-hosted JSON file named `<repo-name>.json` — e.g. `https://img.shields.io/endpoint?url=https://gist.githubusercontent.com/bsubhamay/<gist-id>/raw/terraform-snowflake-view.json`. If the `<repo-name>.json` file does not exist in a gist yet, create a new secret gist (e.g. `gh gist create <file> --desc "<repo-name>.json"`) and update the badge URL via `utils/update-badge.sh`; any leftover reference to a prior template's JSON file (e.g. `terraform-aws-dynamodb.json`) must be replaced
  - All **Usage** HCL snippets — `source = "github.com/subhamay-bhattacharyya-tf/terraform-snowflake-view?ref=main"` must point at the repo root (no `/modules/<name>` path) and match the current repo name; example inputs must use this module's variable shape (`views`), not leftover template shapes (`tables`, `autoscaling`, `s3_config`, etc.)
  - The **Examples** table — entries must match the directories under `examples/` (`basic`, `secure-view`)
  - **Inputs** / **Outputs** / **Resources Created** tables — must match the fields declared in `variables.tf` / `outputs.tf` and the resources in `main.tf`; regenerate via `utils/generate-docs.sh`
  - **Validation**, **Testing**, and **CI/CD** sections — test commands must reference the `tests/` directory (plural), and CI job names and required secrets/variables (`SNOWFLAKE_ACCOUNT`, `SNOWFLAKE_USER`, `SNOWFLAKE_PRIVATE_KEY`, `SNOWFLAKE_ROLE`, `SNOWFLAKE_WAREHOUSE`) must reference Snowflake views, not leftover services
  - **All GFM tables in `README.md` must be pipe-aligned (markdownlint rule MD060, `table-column-style: aligned`).** After any edit that adds or changes a table, re-align every table so the `|` characters line up vertically across the header, the separator row, and all body rows. Run `utils/align-md-tables.py` to enforce this automatically — do not leave mixed-width tables behind; MD060 will flag them in the IDE. A quick structural check: every row of a given table must have its `|` characters at the same string-column positions (this can be verified by scripting `[k for k,c in enumerate(row) if c=='|']` and asserting the list is identical for every row in the table)
  - **Every heading in `README.md` must be unique across the whole document (markdownlint rule MD024, `no-duplicate-heading`), regardless of heading level.** A quick structural check: collect every non-fenced `#{1,6}` heading and assert `collections.Counter(texts)` has no value `> 1`
- `.github/workflows/ci.yaml` must always reflect the Snowflake object being provisioned by this module — job names, step descriptions, test targets (which must point to `tests/`, not `test/`), and required secrets/variables must reference Snowflake views (not leftover references from the upstream template such as DynamoDB, S3, GCS, or other services)

All validation (view type values, secure flag handling, required SQL `statement`, database/schema name format, view name format, required fields) lives in `variables.tf`.

## CI pipeline (`.github/workflows/ci.yaml`)

Runs on pushes/PRs to `main`, `feature/**`, `bug/**` when root-level `*.tf` files, `examples/**`, or `tests/**` files change. All job names, step descriptions, and environment variables in this workflow must reference Snowflake views — any leftover references to other services (DynamoDB, S3, GCS, IAM, etc.) from the upstream template must be replaced. The default Terraform version is pinned to `1.5.0` via the `TF_VERSION` env (overridable through the `TERRAFORM_VERSION` repo variable) to satisfy `versions.tf`'s `required_version = ">= 1.5.0"`. The default Go version is pinned to `1.22` via `GO_VERSION` (overridable through the `GO_VERSION` repo variable) to match `tests/go.mod`.

1. **terraform-validate** — `fmt -check`, `init`, `validate` on the root module
2. **examples-validate** — `init` + `validate` on every example under the `example` matrix (`examples/basic`, `examples/secure-view`) (needs step 1)
3. **terratest** (`Snowflake View Terratest`) — real Snowflake integration test from `tests/` (needs step 2); runs `go test -run TestSnowflakeViewBasic` and reads the `SNOWFLAKE_ACCOUNT`, `SNOWFLAKE_USER`, `SNOWFLAKE_PRIVATE_KEY`, `SNOWFLAKE_ROLE`, `SNOWFLAKE_WAREHOUSE` repo secrets plus the `SNOWFLAKE_TEST_DATABASE` and `SNOWFLAKE_TEST_SCHEMA` repo variables
4. **generate-changelog** — runs `git-cliff` on non-main branches (needs step 2)
5. **semantic-release** — runs only on `main` after steps 2 and 3; uses Conventional Commits and the inline `extra_plugins` block to auto-version

Local-only gates (not invoked from CI in the current workflow): `utils/lint.sh` (tflint + trivy), the docs-drift check (`utils/generate-docs.sh` + `utils/align-md-tables.py` followed by `git diff --exit-code README.md`), and `utils/update-badge.sh`. Run these via `pre-commit` before pushing until the corresponding CI jobs are added back.

## Commit message convention

Follows **Conventional Commits** — semantic-release uses this to determine the next version:

- `feat:` → minor bump
- `fix:` → patch bump
- `chore:`, `docs:`, `refactor:`, etc. → no release
- Breaking changes via `BREAKING CHANGE:` footer → major bump
