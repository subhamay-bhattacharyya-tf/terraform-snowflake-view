# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this module does

This is a **Terraform module** that creates and manages **Snowflake views** ‚Äî both standard and secure ‚Äî using infrastructure as code.

The module is a single, flat Terraform module rooted at the repository root (`main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`). It accepts a single map-based input (`views`) and exposes maps of resource attributes as outputs, allowing one module call to provision many views via `for_each`.

The module manages only `snowflake_view` resources. Grants and other access-control concerns are intentionally out of scope and should be managed by separate, composable modules to prevent drift and align with Snowflake's privilege model.

## Examples

The `examples/` directory contains end-to-end, runnable configurations that exercise the module against real Snowflake views. Each example is a self-contained Terraform configuration that calls the root module via a relative `source = "../.."` reference, declares its own `terraform` and `provider` blocks, and uses the module's `views` map input.

Each example must:

- Use the module's `views` map-based input shape ‚Äî never inline `snowflake_view` resources
- Reference real Snowflake objects (database, schema, base tables) that a Terratest run can create or assume exist
- Include a short `README.md` explaining what the example demonstrates and the Snowflake objects it provisions
- Be validated by the `examples-validate` CI step (`terraform init -backend=false && terraform validate`)

### `basic/`

A single **standard view** built over a base table ‚Äî the minimum viable usage. Demonstrates:

- Calling the module with a single-entry `views` map
- Setting `is_secure = false` (or omitting it to use the default)
- Providing a SQL `statement` that selects from a base table
- Wiring `database` and `schema` inputs to a target Snowflake schema

Use this as the reference for new users learning the module's input shape.

### `secure-view/`

A **secure view** for restricted data exposure ‚Äî demonstrates the security-oriented use case. Demonstrates:

- Setting `is_secure = true` to mark the view as secure (hiding the underlying SQL definition from non-owners)
- Selecting a curated subset of columns suitable for downstream consumers
- Adding a meaningful `comment` describing why the view is secure and which roles should consume it

Use this as the reference when exposing PII, financial, or otherwise sensitive data through views.

#### Sample `views` map

```hcl
module "snowflake_view" {
  source = "../.."

  views = {
    customer_pii_secure = {
      name      = "CUSTOMER_PII_SECURE_VW"
      database  = "ANALYTICS_DB"
      schema    = "CONSUMPTION"
      is_secure = true

      statement = <<-SQL
        SELECT
          cust_key,
          name,
          nation_name,
          mkt_segment
        FROM ANALYTICS_DB.CLEAN.CUSTOMER_CLEAN_DT
      SQL

      comment = "Secure view exposing curated customer attributes; consumed by ANALYST_ROLE only. Underlying SQL hidden from non-owners."
    }
  }
}
```

The map key (`customer_pii_secure`) is a logical Terraform identifier used in `for_each` and in output maps; the actual Snowflake view name (`CUSTOMER_PII_SECURE_VW`) lives inside the object as the `name` field. This separation is intentional ‚Äî it lets the same Terraform identifier survive a Snowflake-side rename without forcing a destroy/create.

## Utility Scripts

The `utils/` directory contains helper scripts used by both local development and CI. Anything reusable across `pre-commit`, `make` targets, and GitHub Actions belongs here ‚Äî not at the repo root and not duplicated inside individual workflows.

All scripts must:

- Be runnable from the repository root (`bash utils/<script>.sh` or `python utils/<script>.py`)
- Be idempotent ‚Äî running them twice in a row produces no diff on a clean tree
- Exit non-zero on failure so they can gate CI steps and pre-commit hooks
- Reference Snowflake views in any user-facing output (no leftover references to DynamoDB, S3, etc.)

### `generate-docs.sh`

Runs `terraform-docs` against the root module and rewrites the auto-generated **Inputs**, **Outputs**, and **Resources Created** tables in `README.md`. Should be invoked:

- Locally before committing any change to `variables.tf`, `outputs.tf`, or `main.tf`
- By the `pre-commit` hook chain
- By a CI doc-drift check that fails the build if `README.md` is out of sync with the module

After this script runs, `align-md-tables.py` should be run to re-pipe-align any tables that `terraform-docs` regenerated.

### `lint.sh`

Wraps **tflint** and **trivy** to lint the root Terraform module and scan it for security issues. Runs:

- `tflint --recursive` with the project's `.tflint.hcl` configuration ‚Äî catches Snowflake provider misuse, unused variables, deprecated syntax
- `trivy config .` ‚Äî scans for misconfigurations (e.g. views that should be secure but aren't, missing comments on production resources)

The script exits non-zero on any finding so it can gate CI. Both tools are installed by `install-tools.sh`.

### `align-md-tables.py`

Python auto-aligner that pads every GFM table cell in `README.md` to its column's maximum width, enforcing **markdownlint MD060** (`table-column-style: aligned`). Should be invoked:

- Whenever `generate-docs.sh` regenerates inputs/outputs tables
- Whenever a heading or table is hand-edited in `README.md`
- By the `pre-commit` hook chain so misaligned tables never reach CI

The script collects every `|` position per row and asserts they line up vertically across the header, separator, and body rows ‚Äî the same structural check described under the README conventions.

### `update-badge.sh`

Updates the shields.io custom-endpoint gist JSON file (`terraform-snowflake-view.json`) hosted as a secret gist. Used to keep the README's coverage / test-status / version badge in sync with the latest CI run. Should be invoked:

- By the `terratest` CI job after a successful Snowflake views integration test
- By the `semantic-release` job after a new version is published

Requires `gh` CLI authentication and the gist ID configured via the `BADGE_GIST_ID` environment variable.

## Tests

The `tests/` directory contains **Terratest** integration tests written in Go that exercise the module against a real Snowflake account. These are not unit tests ‚Äî they `terraform apply` real `snowflake_view` resources, assert behavior, and `terraform destroy` on teardown.

Each test file must:

- Live under `tests/` (note: plural) ‚Äî never `test/`
- Be executable via `go test` from the `tests/` directory
- Use the helpers defined in `helpers_test.go` rather than re-implementing setup/teardown
- Target one of the configurations under `examples/` as its Terraform working directory
- Clean up all Snowflake objects in `defer terraform.Destroy(...)` ‚Äî failures must not leak views into the target account
- Reference Snowflake views in test names, log lines, and assertion messages (no leftover references to DynamoDB, S3, etc.)

Required environment variables (provided as repo secrets/vars in CI):

- `SNOWFLAKE_ACCOUNT` ‚Äî target Snowflake account identifier
- `SNOWFLAKE_USER` ‚Äî service-account user with privileges to create views in the test database
- `SNOWFLAKE_PRIVATE_KEY` ‚Äî key-pair auth private key (preferred over passwords)
- `SNOWFLAKE_ROLE` ‚Äî role used for the test session; must have `CREATE VIEW` on the target schema
- `SNOWFLAKE_WAREHOUSE` ‚Äî warehouse used to compile view definitions during apply

### `snowflake_view_basic_test.go`

End-to-end test for the `examples/basic/` configuration. Covers:

- **Apply** ‚Äî runs `terraform init` + `terraform apply` against `examples/basic/` and asserts the apply succeeds with zero errors
- **Output assertions** ‚Äî reads the module's output maps (`view_ids`, `view_fully_qualified_names`, `view_names`, `is_secure`) and asserts each map has exactly the expected keys and that the values are non-empty
- **Snowflake-side verification** ‚Äî connects directly to Snowflake and asserts the view exists in `INFORMATION_SCHEMA.VIEWS`, that `IS_SECURE = 'NO'` for the standard view, and that the resolved `view_definition` contains the expected `SELECT` columns
- **Idempotency** ‚Äî runs `terraform plan` after apply and asserts the plan is empty (no drift on a no-op re-run)
- **Destroy** ‚Äî runs `terraform destroy` in a `defer` block and asserts the view no longer appears in `INFORMATION_SCHEMA.VIEWS`

This test is the reference for what a "passing" module change looks like ‚Äî any new feature should be exercised by either extending this test or adding a sibling test file alongside a new example.

### `helpers_test.go`

Shared helpers consumed by every test file under `tests/`. Provides the contract that every Snowflake-views test relies on:

- **`buildTerraformOptions(t, exampleDir)`** ‚Äî constructs a `*terraform.Options` pointed at the given example directory, wired with the `SNOWFLAKE_*` environment variables and a unique per-run resource suffix so parallel CI runs don't collide on view names
- **`newSnowflakeClient(t)`** ‚Äî opens an authenticated `*sql.DB` connection to Snowflake using the same credentials Terraform uses, scoped to the test role and warehouse, with a context-aware timeout
- **`assertViewExists(t, db, fullyQualifiedName)`** ‚Äî queries `INFORMATION_SCHEMA.VIEWS` and asserts the view is present; fails fast with a descriptive message if not
- **`assertViewIsSecure(t, db, fullyQualifiedName, expected bool)`** ‚Äî asserts the `IS_SECURE` flag matches the expected value, used to verify the `is_secure` input is honored end-to-end
- **`assertViewDestroyed(t, db, fullyQualifiedName)`** ‚Äî asserts the view is absent from `INFORMATION_SCHEMA.VIEWS`, used in the post-destroy phase to catch leaked resources
- **`uniqueSuffix(t)`** ‚Äî returns a short, lowercase, alphanumeric suffix derived from the test name and a random seed; appended to view and database names to keep parallel runs isolated

Helpers must not embed test logic ‚Äî they are pure setup, teardown, and assertion primitives. Anything specific to a single test (e.g., asserting a particular column list) belongs in that test file, not here.

## Key Conventions

- The module follows the standard flat Terraform layout (`main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`) at the repository root ‚Äî there are no submodules under `modules/`
- All infrastructure changes go through Terraform ‚Äî never modify Snowflake views manually via the UI or worksheets
- The module accepts a single `map(object({...}))` input named `views`, consumed via `for_each`
- The map key is a logical Terraform identifier; the actual Snowflake view name is a field inside the object
- Grants are intentionally **not** managed by this module ‚Äî pair this module with a separate grants module to align with Snowflake's privilege model and avoid drift
- Every example under `examples/` must call the root module via `source = "../.."` and use the `views` map input ‚Äî no inline `snowflake_view` resources
- Helper scripts live under `utils/` ‚Äî anything reusable across local dev and CI (doc generation, lint wrappers, validation helpers, badge updates) belongs there, not at the repo root
- Integration tests live under `tests/` (note: plural), not `test/` ‚Äî Go test files, helpers, and any test fixtures all live in this directory; helpers are pure setup/teardown/assertion primitives, never test logic
- `package.json` and `package-lock.json` must always have their `name` field set to the current repository name (`terraform-snowflake-view`)
- `CONTRIBUTING.md` must always reference the current repository (`terraform-snowflake-view`), including the **Reporting Issues** section which must link to this repository's issues page
- `README.md` must always reflect the current repository (`terraform-snowflake-view`) and the Snowflake object this module provisions (Snowflake views). Every surface that carries the repo name or service name must be kept in sync ‚Äî specifically:
  - Title (`# Terraform Snowflake View Module`) and the one-line description below it
  - All shields.io badges that interpolate the repo slug (`subhamay-bhattacharyya-tf/terraform-snowflake-view`) ‚Äî leftover slugs from the upstream template (e.g. `terraform-aws-s3`, `terraform-aws-dynamodb`) must be swapped
  - The shields.io custom-endpoint badge, which points to a gist-hosted JSON file named `<repo-name>.json` ‚Äî e.g. `https://img.shields.io/endpoint?url=https://gist.githubusercontent.com/bsubhamay/<gist-id>/raw/terraform-snowflake-view.json`. If the `<repo-name>.json` file does not exist in a gist yet, create a new secret gist (e.g. `gh gist create <file> --desc "<repo-name>.json"`) and update the badge URL via `utils/update-badge.sh`; any leftover reference to a prior template's JSON file (e.g. `terraform-aws-dynamodb.json`) must be replaced
  - All **Usage** HCL snippets ‚Äî `source = "github.com/subhamay-bhattacharyya-tf/terraform-snowflake-view?ref=main"` must point at the repo root (no `/modules/<name>` path) and match the current repo name; example inputs must use this module's variable shape (`views`), not leftover template shapes (`tables`, `autoscaling`, `s3_config`, etc.)
  - The **Examples** table ‚Äî entries must match the directories under `examples/` (`basic`, `secure-view`)
  - **Inputs** / **Outputs** / **Resources Created** tables ‚Äî must match the fields declared in `variables.tf` / `outputs.tf` and the resources in `main.tf`; regenerate via `utils/generate-docs.sh`
  - **Validation**, **Testing**, and **CI/CD** sections ‚Äî test commands must reference the `tests/` directory (plural), and CI job names and required secrets/variables (`SNOWFLAKE_ACCOUNT`, `SNOWFLAKE_USER`, `SNOWFLAKE_PRIVATE_KEY`, `SNOWFLAKE_ROLE`, `SNOWFLAKE_WAREHOUSE`) must reference Snowflake views, not leftover services
  - **All GFM tables in `README.md` must be pipe-aligned (markdownlint rule MD060, `table-column-style: aligned`).** After any edit that adds or changes a table, re-align every table so the `|` characters line up vertically across the header, the separator row, and all body rows. Run `utils/align-md-tables.py` to enforce this automatically ‚Äî do not leave mixed-width tables behind; MD060 will flag them in the IDE. A quick structural check: every row of a given table must have its `|` characters at the same string-column positions (this can be verified by scripting `[k for k,c in enumerate(row) if c=='|']` and asserting the list is identical for every row in the table)
  - **Every heading in `README.md` must be unique across the whole document (markdownlint rule MD024, `no-duplicate-heading`), regardless of heading level.** A quick structural check: collect every non-fenced `#{1,6}` heading and assert `collections.Counter(texts)` has no value `> 1`
- `.github/workflows/ci.yaml` must always reflect the Snowflake object being provisioned by this module ‚Äî job names, step descriptions, test targets (which must point to `tests/`, not `test/`), and required secrets/variables must reference Snowflake views (not leftover references from the upstream template such as DynamoDB, S3, GCS, or other services)

All validation (view type values, secure flag handling, required SQL `statement`, database/schema name format, view name format, required fields) lives in `variables.tf`.

## CI pipeline (`.github/workflows/ci.yaml`)

Runs on pushes/PRs to `main`, `feature/**`, `bug/**` when root-level `*.tf` files, `examples/**`, or `test/**` files change. All job names, step descriptions, and environment variables in this workflow must reference Snowflake views ‚Äî any leftover references to other services (DynamoDB, S3, GCS, IAM, etc.) from the upstream template must be replaced. The default Terraform version is pinned to `1.5.0` via the `TF_VERSION` env (overridable through the `TERRAFORM_VERSION` repo variable) to satisfy `versions.tf`'s `required_version = ">= 1.5.0"`.

1. **terraform-validate** ‚Äî `fmt -check`, `init`, `validate` on the root module
2. **examples-validate** ‚Äî `init` + `validate` on every example under the `example` matrix (`examples/basic`, `examples/secure-view`) (needs step 1)
3. **terratest** ‚Äî real Snowflake integration test from the test working directory (needs step 2); requires `SNOWFLAKE_ORGANIZATION_NAME`, `SNOWFLAKE_ACCOUNT_NAME`, `SNOWFLAKE_USER`, `SNOWFLAKE_ROLE` repo vars and `SNOWFLAKE_PRIVATE_KEY` repo secret; runs two test entrypoints (`TestSingleWarehouse` and `TestMultipleWarehouses`) ‚Äî these names must be re-aligned to the Snowflake-views Go entrypoints under `tests/` before the job will pass
4. **generate-changelog** ‚Äî runs `git-cliff` on non-main branches (needs step 2)
5. **semantic-release** ‚Äî runs only on `main` after steps 2 and 3; uses Conventional Commits and the inline `extra_plugins` block to auto-version

Notes about the current on-disk shape vs. the intended Snowflake-views shape: the workflow no longer invokes `utils/lint.sh` (tflint + trivy) inside `terraform-validate`, no longer runs a `docs-drift` job, and no longer calls `utils/update-badge.sh` after `terratest` or `semantic-release`. The `terratest` job's path filter (`test/**`), `working-directory: test`, test selectors (`TestSingleWarehouse` / `TestMultipleWarehouses`), and env var names (`SNOWFLAKE_ORGANIZATION_NAME`, `SNOWFLAKE_ACCOUNT_NAME`) are leftovers from the upstream warehouse template ‚Äî the actual Go code lives under `tests/` (plural) with a single `TestSnowflakeViewBasic` entrypoint that reads `SNOWFLAKE_ACCOUNT` / `SNOWFLAKE_USER` (and the additional vars listed under the **Tests** section above). Keep this divergence list in mind when editing the workflow or restoring the lint / docs-drift / badge-refresh jobs.

## Commit message convention

Follows **Conventional Commits** ‚Äî semantic-release uses this to determine the next version:

- `feat:` ‚Üí minor bump
- `fix:` ‚Üí patch bump
- `chore:`, `docs:`, `refactor:`, etc. ‚Üí no release
- Breaking changes via `BREAKING CHANGE:` footer ‚Üí major bump
